---
title: Tokens & claims
description: The access and refresh credentials returned by NamoID Hosted Auth and how to handle them safely.
---

Exchanging a Hosted Auth code returns an **access token** and, for a
confidential integration, a **refresh token**.

```json
{
  "access_token": "…",
  "refresh_token": "…",
  "token_type": "Bearer",
  "expires_in": 900,
  "user_id": "…"
}
```

The exact `expires_in` value comes from the environment's server configuration;
clients should use the returned value instead of assuming a lifetime.

## Access token

The access token is a short-lived signed token for NamoID Auth APIs. Relevant
claims include:

| Claim | Meaning |
| --- | --- |
| `sub` | Stable NamoID user ID |
| `sid` | Native session ID |
| `jti` | Unique token identifier |
| `token_use` | `access` |
| `auth_mode` | `native` |
| `tid`, `pid`, `eid` | Tenant, project, and environment IDs |
| `email` | Current email when present |
| `exp`, `iat` | Expiry and issue timestamps |

Send it as `Authorization: Bearer <access_token>`. A trusted backend can use
`validateAuthToken` from `@namoidhq/js/server`, which validates it through the
NamoID Auth API with an Auth secret key.

## Refresh token

Confidential Hosted Auth exchanges return a refresh token. It rotates on every
use: save the new value and discard the one you sent. Reusing a previous token
is treated as replay and revokes the session.

Public browser exchanges intentionally omit refresh tokens because a browser
cannot protect a durable bearer credential.

## Storage

- Store tokens only on a trusted server when possible.
- Use an encrypted server-side session store and an `HttpOnly` application
  cookie.
- Never log tokens or include them in URLs.
- Do not persist refresh tokens in browser storage.
- Revoke the native session during logout.
