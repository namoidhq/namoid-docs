Every project has isolated **Test** and **Live** environments. Build and verify
your integration in Test without adding users to production or consuming the
Live OTP budget.

## What is isolated

Each environment has its own:

- users and sessions
- Auth publishable and secret keys
- Hosted Auth URL, callbacks, origins, and branding
- authentication methods, MFA policy, and access mode
- OTP budget, webhooks, and audit data

## Environment URLs

```text
Test: https://<slug>.sandbox.namoid.in
Live: https://<slug>.id.namoid.in
```

Always copy the exact Hosted Auth URL from the selected environment rather than
constructing it yourself.

## Credential prefixes

| Credential | Test | Live |
| --- | --- | --- |
| Auth publishable key | `namoid_auth_pk_test_…` | `namoid_auth_pk_live_…` |
| Auth secret key | `namoid_auth_sk_test_…` | `namoid_auth_sk_live_…` |

Platform keys are tenant-scoped and use `namoid_platform_sk_…`. The target
project and environment are identified by the management API route.

## OTP spend caps

Each environment has a monthly OTP spend limit. Sends reserve budget before
delivery and refund it if delivery fails. Further sends are blocked when the
environment reaches its cap.

## Promote configuration

Use the promote-to-Live flow to review and copy authentication configuration
from Test. Users and credentials are never copied, so create new Live auth keys
and configure the production callback URL separately.

During development, use a Test key and the Test Hosted Auth URL. At launch,
switch both values to their Live equivalents.
