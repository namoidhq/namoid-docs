---
title: Sessions
description: How NamoID Hosted Auth sessions, application sessions, logout, refresh, and revocation work.
---

A NamoID session represents an authenticated user and device in one
environment. It is separate from your application's own session.

## Session lifecycle

- Hosted Auth establishes a session after the user completes authentication.
- Exchanging the one-time callback code returns a short-lived access token.
- Confidential integrations also receive a refresh token. Public browser
  integrations do not receive a durable refresh token.
- Refresh rotates the refresh token; only the newest value should be retained.
- Logout, account security changes, expiry, or administrative revocation ends
  the corresponding session.

Session records are persisted and can be listed or revoked from account pages
and authenticated API endpoints.

## Application sessions

For a server-rendered application, exchange the Hosted Auth code on the server
and store credentials in a server-side session. Give the browser only an
`HttpOnly`, `SameSite=Lax`, `Secure` application cookie.

Avoid placing access or refresh tokens in `localStorage`. The
[Next.js SDK](/sdks/nextjs) handles the Hosted Auth transaction in `HttpOnly`
cookies and lets your callback persist tokens in your own session store.

## Logout

A complete logout has two parts:

1. Clear your application's local session.
2. Revoke the NamoID native session and visit `/sign-out` on the environment
   host to clear its browser session.

The Next.js SDK's `logout()` helper performs the NamoID portion. Pass the access
and refresh token when available, and clear your own session before returning
the redirect response.

## Revocation and replay protection

Refresh tokens rotate on use. Reusing an older refresh token is treated as a
replay and revokes the session. Users and administrators can also revoke an
individual session or all sessions for an account.
